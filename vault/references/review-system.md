# Review System Specification

## 目的
- ものづくり補助金／成長加速化補助金の事業計画書レビューを、再現可能なマルチエージェント手順で標準化する。
- 根拠付き評価を必須化し、スコア・指摘・改善提案の一貫性を上げる。

## 対象と前提
- 対象入力: 事業計画書本文、公募要領、審査基準、数値計画（Excel/表計算）、参考資料。
- 運用基盤: Obsidian Vault + GitHub + Codex（統括）+ Claude Code（実行）。
- レビュー結果は `vault/RUN/` に保存し、参照元パスを残す。

## 3つの運用目的
1. 事業計画書生成システム
- Obsidian + VSCode + Claude Code で計画書品質を継続改善する。

2. マルチエージェント審査フレームワーク
- フォーマル版（8体制）と実践版（4体制）を使い分ける。

3. スキルファイルベースレビュー
- `SKILL.md` と `vault/references/` のルールを根拠に採点とFBを行う。

## フォーマル版（8体制）
### Phase 0: ゲート
- ECA-01（適格性審査）
- 判定: Pass / Fail
- ルール: Fail の場合は以降を停止（Critical Failure）。

### Phase 1: 並列評価
- TRA-01: 技術・革新性・実現可能性（30点）
- CRA-01: 事業化・市場・競争優位・BM（40点枠の主部）
- FAA-01: 財務・付加価値・賃金・回収計画（40点枠の一部）
- MAA-01: 市場データ品質・競合分析・トレンド根拠
- PAA-01: 政策適合・加点・地域波及・先端性（20点）

補足（配点運用）:
- 100点に正規化するため、事業化40点枠は CRA+FAA の合算で扱う。
- 既定案: CRA 25点 + FAA 15点（必要に応じて公募回ごとに見直し）。

### Phase 2: 統合
- CVA-01: 整合性検証（論理・数値・記載品質・矛盾検知）
- MRA-01: 統括判定（総合スコア0-100、グレードA-E、改善優先順位）

## 実践版（4体制）
- 現場の短時間レビュー向けに、観点を4役へ集約する。
- 例: 公正審査 / 批判審査 / 好意審査 / 専門審査。
- フォーマル版に移行する前段の壁打ち用途とする。

## 各エージェントの入出力
### 共通入力
- 計画書本文（Markdown/PDF/Docx化テキスト）
- 公募要領・審査基準
- 数値計画（売上、利益、付加価値、賃金、投資、回収）
- 参考根拠（市場統計、競合情報、政策文書）

### 共通出力
- スコア（該当エージェントのみ）
- 指摘一覧（重要度付き: Critical/High/Medium/Low）
- 改善提案（実行可能な修正指示）
- 根拠リスト（参照パス・節・表番号）

## 受入基準（レビュー運用）
1. 根拠必須
- すべての評価項目に出典または参照パスを付与する。

2. 推測禁止
- 根拠未確認の断定を行わない。未確認は「要確認」として分離記載。

3. 追跡可能性
- 指摘は、対象箇所（章/段落/表）を指定する。

4. 再現可能性
- 評価手順と判断理由が、別担当者でも再現できる粒度で残っている。

## CVA -> MRA の矛盾処理
1. CVA が矛盾を検知
- 例: 本文KPIと数値計画の不一致、要件未達、相互に矛盾する主張。

2. CVA が矛盾チケットを発行
- フィールド: `id`, `description`, `location`, `affected_agents`, `severity`, `required_fix`。

3. MRA が最終判定で反映
- Critical 矛盾が未解消なら Grade 上限を制限（例: C以下）。
- 未解消数に応じて総合点を減点し、改善優先順位を再計算。

## Critical Failure 条件
- ECA-01 が Fail。
- 公募要件の必須条件（申請枠適合、基本要件、対象経費など）に重大欠落。
- 主要数値の整合不能（本文と計画値が矛盾し回復不能）。
- 根拠不在のまま主要主張を維持している。

## 運用ルール
- 真実の置き場は本ファイルと `agents/*.yml`。
- 新ルール追加は、まず本ファイルか manifest を更新して差分管理する。
- プロンプトだけでルールを増やさない。
